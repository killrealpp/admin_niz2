# 2026-05-28 - best2info как клиентская база знаний

## Решение

Ввести `best2info/` как отдельную markdown-wiki для клиентских информационных ответов.

`best2obs/` остается памятью разработки и архитектуры. `best2info/` хранит факты, которые бот может говорить клиенту: объекты, вместимость, цены, скидки, оплата, локация, дети/животные, правила отдыха и допы.

## Причина

Один большой knowledge markdown смешивал разные типы фактов и плохо контролировал routing. В live-чате 6093 это проявилось так: info-вопросы про скидки не доходили до фактов, а availability/анкета иногда воспринимали короткие сообщения не по текущему шагу.

Новая схема закрепляет правило проекта:

- AI определяет смысл и intent.
- Backend валидирует состояние и решает маршрут.
- `answer_info` получает только релевантные chunks из `best2info`.
- `check_availability` всегда проверяет локальную БД/YCLIENTS-cache.
- Booking form patch применяется только state-safe: по текущему шагу или по явно сказанному клиентом полю.

## Следствия

- `app/knowledge` временно остается legacy/fallback и не удаляется сразу.
- Для новых клиентских фактов сначала обновлять `best2info`, а не промпт и не development wiki.
- Нельзя чинить info-ответы костылями по одному слову, если факт должен жить в базе знаний. Ключевые слова допустимы только как fallback/retrieval boost.
- Для availability нельзя использовать `best2info` как источник свободности; свободность берется только из backend availability.
